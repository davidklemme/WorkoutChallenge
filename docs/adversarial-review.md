# Adversarial Review: WorkoutChallenge Spec

## Purpose

This review challenges the first product spec from the perspective of product risk, implementation speed, privacy, motivation design, and long-term maintainability. It is intentionally skeptical so the next iteration can make sharper decisions before building.

## Executive summary

The current spec is directionally strong: personal-first, mobile-first, local-first, encouraging, and focused on daily completion through short sets. The main gap is that it still leaves several foundational product decisions open. Those decisions will affect the data model, MVP shape, and whether the first build feels immediately useful or too generic.

The biggest risk is trying to support too many active movements, units, rest states, plan types, and history concepts before the core loop is proven. The fastest path to a usable product is a deliberately narrow MVP: a small exercise library, one active daily training session, one local device, one Today screen, simple history, and no social/account features.

## What is missing from the current spec

### 1. A precise MVP decision set

The spec lists scope and open questions, but it does not yet make enough hard calls for implementation. Without these calls, the first build can drift.

Missing decisions:

* Is the MVP one active exercise per day or multiple active exercises?
* What is the first exercise library: push-ups, planks, squats, and custom?
* Are daily targets manually set, repeated from exercise defaults, or generated by a monthly plan?
* Does a new day auto-create from defaults?
* What happens when the app is opened after missed days?
* Does the MVP support rest days at launch?
* Which persistence layer is first: `localStorage` or IndexedDB?
* What is the minimum export format?
* Are notifications intentionally excluded from MVP?

Recommended stance:

* Use one active exercise/training session per day for MVP.
* Seed push-ups, planks, and squats, while allowing custom exercises.
* Auto-create each day from the user's default exercise goal or current monthly plan phase.
* Infer missed days only when rendering history, not by backfilling many records.
* Include manual rest days only if cheap; otherwise defer.
* Start with `localStorage` behind a repository interface, then migrate to IndexedDB if needed.
* Export JSON first; CSV later.
* Exclude notifications until the core loop proves useful.

### 2. A sharper definition of “usable fast”

The phrase is good, but the spec needs measurable launch criteria.

Suggested launch bar:

* A new user can set up a goal and log the first set in under 60 seconds.
* Returning users can log a set in under 10 seconds.
* Today’s progress survives refresh and browser restart.
* The app can be installed as a PWA.
* The app remains useful offline after initial load.
* History shows at least the last 14 days.

### 3. Explicit non-goals

The spec says what is out of scope, but it should be more forceful about what the app must not become.

Non-goals to state clearly:

* No public leaderboard.
* No default social feed.
* No streak shaming.
* No forced account creation.
* No paywall around core tracking or history.
* No medical claims.
* No algorithmic coaching in MVP.

### 4. Health and safety boundaries

The long-term health focus is a strength, but the spec needs a safety boundary. A fitness app can accidentally encourage overtraining or inappropriate exercise.

Missing safeguards:

* No medical advice claims.
* Copy should encourage listening to the body.
* Rest should be treated as valid.
* The app should avoid automatically increasing goals without consent.
* Users should be able to lower goals without shame.
* Injury, pain, illness, or fatigue language should direct users to stop and seek appropriate advice.

### 5. A privacy position

Local-first is good, but the spec should make privacy commitments explicit.

Recommended privacy stance:

* No account required for MVP.
* No analytics before a consent and privacy policy decision.
* No third-party tracking scripts in MVP.
* All data remains on-device unless the user exports it.
* Sharing, sync, and backup are future opt-in features.

### 6. A clearer naming and identity decision

The repo name is functional, but the product may need a calmer user-facing name. The name will affect copy, manifest metadata, icon direction, and tone.

Options:

* Keep **WorkoutChallenge** for now to avoid delay.
* Use a softer product name later, such as **Daily Bank**, **Move Bank**, **Better Daily**, or **Banked**.

Recommended stance:

* Keep `WorkoutChallenge` as the internal/repo name for MVP.
* Delay brand naming until after the first usable prototype.

### 7. The first screen is not specified enough

The Today screen is listed, but implementation needs a clearer layout hierarchy.

Recommended mobile hierarchy:

1. Date and exercise label.
2. Large progress number: banked / target.
3. Remaining amount.
4. Quick-add buttons.
5. Custom amount.
6. Recent sets.
7. Small link to History and Settings.

The product should avoid hiding the logging controls below too much motivational content. Logging is the primary action.

### 8. History semantics need decisions

History can become surprisingly complex. The spec should define terms now.

Recommended definitions:

* **Complete:** total banked is greater than or equal to target.
* **Partial:** at least one set exists, but total is below target.
* **Missed:** past day with no sets and no rest marker.
* **Rest:** user explicitly marked the day as rest.
* **Future:** never shown as missed.

### 9. Data migration strategy

Even a local-first app needs versioned data. Otherwise early users may lose progress when the schema changes.

Recommended MVP decision:

* Store a `schemaVersion` with persisted data.
* Write storage helpers so migrations can be added later.
* Keep export format versioned.

### 10. Accessibility and inclusivity detail

The spec mentions accessibility but does not yet specify design constraints.

Minimum requirements:

* Large touch targets for quick-add controls.
* Semantic buttons and form labels.
* Visible focus states.
* Good color contrast.
* Progress conveyed by text, not color alone.
* Exercise labels are user-defined so the app is not push-up-only.

## Adversarial questions

### Product questions

1. If the app is private, what makes it motivating after the novelty wears off?
2. Is daily completion too rigid for long-term health?
3. Will users feel worse if they repeatedly miss goals?
4. Does “challenge every day” conflict with recovery?
5. Is the product too generic if it supports any exercise?
6. What is the smallest delightful moment that makes someone return tomorrow?

### UX questions

1. Can the user log a set without reading anything?
2. Can the user correct a mistaken set?
3. Is over-target progress celebrated without encouraging overtraining?
4. Can a user lower the target without feeling like they failed?
5. Is there a clear empty state on day one?
6. Does the app still make sense if used only three days per week?

### Technical questions

1. How will local dates be handled across time zones and daylight saving changes?
2. What happens if the user changes the device clock?
3. How will data be migrated after schema changes?
4. Is `localStorage` enough for MVP, or should IndexedDB be used immediately?
5. How will PWA caching avoid serving stale app code with incompatible local data?
6. How will export/import work if sync is added later?

### Ethical and trust questions

1. Are we accidentally copying too much from Push for Better rather than borrowing the general challenge pattern?
2. Does the app imply mental or physical health benefits it cannot prove?
3. Could streak mechanics create guilt?
4. Will future sharing features undermine the personal-first promise?
5. What data, if any, would ever leave the device?

## Recommended decisions before implementation

These decisions should be made before scaffolding the app.

| Area | Recommended decision | Why |
| --- | --- | --- |
| MVP focus | One active exercise/training session per day | Keeps data model and UI simple. |
| Exercise library | Seed push-ups, planks, and squats, plus custom exercises | Provides a balanced starter set without locking the app to push-ups. |
| Concept hierarchy | Exercise -> Set -> Training Session -> Plan | Gives implementation language that can grow into plans later. |
| Goal source | Daily goal repeats from exercise defaults or current monthly phase | Lets the app auto-create today quickly. |
| Plan shape | Optional monthly plan split into four phases with one break day between phases | Adds structure without building full coaching. |
| Logging | Quick-add presets plus custom set amount | Supports fast sets and flexibility. |
| Editing | Allow deleting mistaken sets | Prevents bad data and user frustration. |
| Rest | Manual rest marker, no scheduling in MVP | Supports health without complex planning. |
| Missed days | Infer in history, do not eagerly create records | Avoids noisy storage and edge cases. |
| Persistence | `localStorage` behind a storage adapter | Fastest build with a migration path. |
| Data shape | Version persisted data from day one | Prevents early schema lock-in. |
| Export | JSON first | Easiest faithful backup. |
| Notifications | Defer | Avoids interruption and permission friction. |
| Analytics | Defer | Maintains trust and avoids policy work. |
| Sharing | Defer | Protects personal-first MVP. |

## Proposed MVP slice

Build only this first:

1. Next.js TypeScript PWA shell.
2. Local storage adapter with versioned data.
3. Exercise library seeded with push-ups, planks, and squats.
4. Create/edit exercise interface for name, unit, default target, and quick-add presets.
5. First-run setup for default exercise, unit, target, and quick-add presets.
6. Today screen as a training session with progress and fast set logging.
7. Delete set action for mistakes.
8. Optional monthly plan setup for one exercise, split into four phases with one break day between phases.
9. Manual complete state when target reached.
10. History list for recent days.
11. Settings to edit defaults and export JSON.

Do not build yet:

* Accounts.
* Cloud sync.
* Social sharing.
* Leaderboards.
* Badges.
* Notifications.
* Multiple active exercises in the same daily session.
* Complex multi-exercise plans.
* Generated coaching plans.

## Biggest risks and mitigations

### Risk: The app becomes a generic habit tracker

Mitigation: Keep the unique interaction centered on banking movement sets toward a daily physical target.

### Risk: The app creates guilt

Mitigation: Avoid streak-first design. Show consistency, but phrase missed days neutrally and make rest valid.

### Risk: The app is too minimal to be motivating

Mitigation: Make the Today screen feel satisfying through clarity, speed, and a calm completion moment rather than badges or social pressure.

### Risk: Local-only data feels fragile

Mitigation: Add JSON export early and design storage with schema versions.

### Risk: Scope creep delays usability

Mitigation: Treat the proposed MVP slice as the build contract. Anything else requires a deliberate follow-up decision.

## Next iteration checklist

Before coding, update the main product spec with these final MVP decisions:

* One active exercise/training session per day.
* Seeded exercises: push-ups, planks, and squats.
* Custom exercise creation.
* Exercise -> set -> training session -> plan hierarchy.
* Optional monthly goal split into four phases with one break day between phases.
* Auto-created daily training session from defaults or current phase.
* Quick-add and custom set logging.
* Set deletion.
* Manual rest marker.
* Inferred missed days.
* Versioned local persistence.
* JSON export.
* No notifications, analytics, accounts, sync, or sharing in MVP.

After that, start implementation with the Today screen and local persistence before adding history or settings.
